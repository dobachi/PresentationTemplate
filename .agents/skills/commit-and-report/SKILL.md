# コミット・プッシュ・Issue報告

引数に続く文字列からコミットメッセージと任意の Issue 番号を読み取る。通常は末尾の数値を Issue 番号、残りをコミットメッセージとみなす。曖昧なら必ず確認する。

## 手順

1. **状況確認**
   - `git status` と `git status -sb` を実行し、現在の変更点を把握して共有する。
2. **ステージングとコミット**
   - 全体をまとめてコミットする前に、必要があれば含めるファイルを確認する。
   - 合意を得たら対象ファイルを明示的にステージングし、指定メッセージでコミットする。
   - コミットが作成されたことを `git log -1 --stat` などで確認し結果を報告する。
3. **プッシュ**
   - 追跡ブランチを確認し、問題なければ `git push` を実行する。
   - 失敗した場合はエラーメッセージを解析し、対処案を提示する。
4. **Issue へのコメント（任意）**
   - Issue 番号が与えられていれば、GitHub CLI が利用可能かを確認する。
   - 利用できる場合は次のようなコメントを投稿する（文章は状況に合わせて調整）:
     ```bash
     gh issue comment <ISSUE_NUM> --body "✅ <コミットメッセージ>"
     ```
   - GitHub CLI が使えない場合は、手動でコメントする手順を案内する。
5. **結果の要約**
   - どのファイルをコミットし、どのリモートにプッシュしたか、Issue へ報告したかを簡潔にまとめる。

## 安全運用ガイド

- 未追跡ファイルを含めるかどうかは都度確認する。
- 既存のコミットを強制的に上書きする操作（`--force` 等）は、明示的な指示があるまで行わない。
- GitHub CLI の認証が済んでいない場合は `gh auth login` を案内し、成功後に再試行する。
